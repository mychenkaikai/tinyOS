- [x] 规格明确以 `x86_64 + QEMU` 作为第一阶段首发目标，而不是同时铺开所有架构
- [x] 规格明确区分 `MPU` 与 `MCU` 两类目标，不把它们当成同一复杂度级别的系统实现
- [x] 规格明确第一阶段范围只覆盖最小启动链路、基础内核能力、简单显示和 GUI MVP
- [x] 规格明确排除完整用户态、POSIX、SMP、GPU 驱动、完整网络栈等高复杂度首阶段目标
- [x] 规格定义了 `boot`、`arch`、`platform`、`kernel` 等核心分层，避免后续实现混杂
- [x] 任务列表覆盖了启动、内核、显示输入、GUI、跨架构预留、MCU 子集和验证路径
- [x] 任务顺序体现依赖关系，不存在先做 GUI 再补启动链或先做 MCU 再补抽象边界的倒挂
- [x] 规格给出了第二阶段 `ARM64 virt` 与 `RISC-V virt` 的迁移方向，而不是只停留在 x86 想象
- [x] 规格给出了 MCU 子集的收敛方式，而不是空泛地声称“以后支持 MCU”
- [x] 整套计划的里程碑都具备可验证的用户可见结果，而不是只有内部技术动作
